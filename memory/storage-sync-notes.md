# 数据库与存储桶同步学习记录

## 1. 为什么会出现存储桶残留文件

本项目的视频信息分开保存在两个地方：

```txt
PostgreSQL：保存标题、介绍、视频 URL、封面 URL
MinIO / COS：保存真正的视频文件和封面图片
```

这两个系统是独立的。

所以：

```txt
删除数据库记录，不会自动删除存储桶文件
删除存储桶文件，也不会自动删除数据库记录
```

必须由后端代码主动维护它们的一致性。

## 2. 什么是孤儿文件

孤儿文件指：

```txt
存储桶里存在文件
但是数据库 videos 表里已经没有任何记录引用它
```

例如：

```txt
MinIO 里有 videos/a.mp4
但是数据库里没有任何 video_url 指向它
```

这个文件就叫孤儿文件。

## 3. 为什么之前会有 5 个视频文件

用户一共上传过 5 条视频，后来删除过 2 条。

数据库当前只剩 2 条记录，但存储桶里还有 5 个视频文件，说明至少有一部分文件没有被同步删除。

常见原因：

```txt
1. 删除功能完善前上传/删除过数据
2. 删除接口先删数据库，后删存储桶，中途存储桶删除失败
3. 本地测试时多次切换数据库或存储桶配置
4. 旧代码没有处理存储桶删除
```

## 4. 删除接口的优化

之前删除顺序更容易产生问题：

```txt
1. 删除数据库记录
2. 再删除存储桶文件
```

如果第 1 步成功，第 2 步失败，就会出现：

```txt
数据库记录没了
存储桶文件还在
```

现在改成：

```txt
1. 先 SELECT 查询数据库记录
2. 根据 video_url / cover_url 删除存储桶文件
3. 存储桶删除成功后，再删除数据库记录
```

这样可以减少孤儿文件。

相关文件：

```txt
backend/src/videos.routes.js
```

## 5. 清理孤儿文件脚本

新增脚本：

```txt
backend/scripts/clean-orphan-objects.js
```

它会做这些事：

```txt
1. 查询数据库 videos 表里的 video_url 和 cover_url
2. 转成存储桶 key
3. 列出存储桶 videos/ 和 covers/ 下的所有对象
4. 对比：存储桶有，但数据库没有的 key
5. 默认只预览，不删除
6. 加 --delete 参数才真正删除
```

## 6. 使用方式

### 6.1 先预览孤儿文件

```powershell
npm run storage:clean --prefix backend
```

这个命令只会列出孤儿文件，不会删除。

### 6.2 确认后删除孤儿文件

```powershell
npm run storage:clean --prefix backend -- --delete
```

注意：

```txt
-- --delete
```

中间有两个 `--`。

第一个 `--` 是 npm 用来把参数转交给脚本。

第二个 `--delete` 才是脚本真正收到的参数。

## 8. 学到的知识点

这次练习了：

```txt
数据库和对象存储不是自动同步的
什么是孤儿文件
删除接口的操作顺序很重要
S3 / MinIO 如何列出对象
如何写清理脚本
为什么删除前要先预览
```

## 9. 重要原则

记住：

```txt
数据库保存“文件在哪里”
存储桶保存“文件本体”
```

两者一致性需要后端维护。

删除这类跨系统数据时，尽量：

```txt
先查清楚
先删除外部资源
最后删除数据库记录
必要时提供清理孤儿文件脚本
```
